# Environment Profile Management

This document explains how the HPC Cluster Management System handles user environment profiles using Lmod metamodules, and how the web application automates the entire lifecycle from stack definition to per-user deployment.

---

## 1. Linux Profile Hierarchy — How It Works

When a user logs into a Linux system (SSH or Slurm job), the shell runs a cascade of scripts:

| Script | Scope | When it runs |
|---|---|---|
| `/etc/profile` | All users | Every login |
| `/etc/profile.d/*.sh` | All users | Every login (sourced by `/etc/profile`) |
| `~/.bash_profile` | Single user | SSH / terminal login |
| `~/.bashrc` | Single user | Every new shell / interactive session |

**The Golden Rule:** System-wide configurations (like the Spack module path) belong in `/etc/profile.d/`. User-specific preferences (like which stack to load) go in `~/.bashrc`.

### What `spack_setup.sh` Does
At `/etc/profile.d/spack_setup.sh` (and propagated to compute nodes via the Warewulf `nodeconfig` overlay):
```bash
if [ -f /export/apps/spack/share/spack/setup-env.sh ]; then
    . /export/apps/spack/share/spack/setup-env.sh
fi
module use /export/apps/spack/share/spack/lmod/linux-almalinux9-x86_64/Core
module use /export/apps/custom_modules   # Added by this system
```

## 2. Lmod Metamodules — What They Are

A **metamodule** is a Lua (`.lua`) file that does not represent software itself. Instead, it loads other modules. It acts as a named "profile" that pulls in an entire software stack in a single command.

### File Location
All custom metamodule files are stored on the NFS share:
```
/export/apps/custom_modules/
├── base-developer.lua
├── core-scripting.lua
└── data-utilities.lua
```
This directory is accessible to all nodes via NFS and is added to `$MODULEPATH` by the global profile script.

### Example Lua File
```lua
-- /export/apps/custom_modules/core-scripting.lua
-- Auto-generated by HPC Cluster Management System

help([[Core runtime environments for Python and Perl scripting, database interactions via SQLite, and utilities like JQ.]])
whatis("Version: 1.0")
whatis("Category: Scientific")

load("python/3.14.5-lddvwjv")
load("perl/5.42.0-dadruwd")
load("sqlite/3.53.1-mfudwzt")
load("jq/1.8.1-jnxci33")
```

Once this file exists and the directory is in `$MODULEPATH`, any user can run:
```bash
module load core-scripting
```
And all five modules load in one shot.

## 4. Bashrc Sentinel Block

When a profile is assigned to a user, the backend SSH-injects a managed block into their `~/.bashrc`. The block is delimited by sentinel comments so it can be replaced cleanly on re-assignment without corrupting other user customizations.

```bash
# --- HPC PROFILE MANAGED BLOCK START ---
module purge
module load core-scripting
# --- HPC PROFILE MANAGED BLOCK END ---
```

**Important:** Everything between the sentinel lines is managed by the system. Do not manually edit between these lines — changes will be overwritten on the next profile assignment.

## 5. Web UI — Admin: Environment Profiles Page

Navigate to **Env Profiles** in the admin navigation bar (`/env-stacks`).

### Stack Library (Left Panel)
- Displays all defined stacks as interactive cards.
- **Color coding:** Developer = teal, Scientific = purple, MPI = amber, Custom = pink.
- Click a card to expand and see the generated Lua file content.
- **Edit** (✎) and **Delete** (✕) buttons on each card.
- **"New Stack"** button opens the creation modal.

### Stack Creation Modal
- Fill in: Identifier (slug), Display Name, Category, Description.
- Add module entries one per row. An autocomplete datalist suggests known modules.
- **Live Lua Preview** on the right updates in real-time as you type.
- On submit: the stack is saved to the database and the `.lua` file is written to `/export/apps/custom_modules/` on the master node via SSH.

### User Assignment (Right Panel)
- Table lists all system users with their currently assigned profile.
- Use the dropdown on each row to assign a different stack.
- Selecting "Base Spack (no profile)" resets the user's `~/.bashrc` to the base environment.

## 6. Web UI — User: My Profile Page

Navigate to **My Profile** in the navigation bar (`/my-profile`). Available to **all authenticated users**.

### Choosing a Profile
- Displays all available stacks as selectable option cards.
- Click a card to select it. A checkmark and accent border confirm the selection.
- The **~/.bashrc Preview** section shows exactly what will be written to your login script.
- Click **Apply Profile** to SSH-inject the profile block. Takes effect on your next login.

### Module Explorer
- Select any stack to see its full module list in an animated, numbered list.
- Shows the generated Lua `load()` calls for verification.

### Lmod Collections
Users can also manage **Lmod Collections** — saved snapshots of their own loaded module state:

```bash
# 1. Load whatever you need for your specific project
module load ai-agents
module load gcc/11.2.0

# 2. Save this exact state as a named collection
module save my_dev_env

# 3. Restore it later (instantly purges current env and replaces it)
module restore my_dev_env

# 4. To make a collection load automatically at login
module save default
```

From the UI:
- **Save Collection**: runs `module save <name>` as your user on the master node.
- **Saved Collections**: displays `module savelist` output.
- **Set as Default**: injects `module restore <name>` into `~/.bashrc` as your auto-load command.

## 7. API Reference

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `GET` | `/api/v1/env-stacks/` | Any user | List all stacks |
| `POST` | `/api/v1/env-stacks/` | Admin | Create new stack + write `.lua` |
| `PUT` | `/api/v1/env-stacks/{id}` | Admin | Update stack + regenerate `.lua` |
| `DELETE` | `/api/v1/env-stacks/{id}` | Admin | Delete stack + remove `.lua` |
| `POST` | `/api/v1/env-stacks/{id}/assign/{username}` | Admin | Assign profile to user |
| `DELETE` | `/api/v1/env-stacks/assign/{username}` | Admin | Reset user to base Spack |
| `GET` | `/api/v1/env-stacks/me` | Self | Get own profile + available stacks |
| `POST` | `/api/v1/env-stacks/me/select` | Self | Select own profile |
| `POST` | `/api/v1/env-stacks/me/save-collection` | Self | `module save <name>` |
| `GET` | `/api/v1/env-stacks/me/collections` | Self | `module savelist` |
| `POST` | `/api/v1/env-stacks/me/restore-collection` | Self | Set collection as default in `.bashrc` |

## 8. Troubleshooting

### Profile not loading after login
1. SSH into the master node and check: `cat /home/<username>/.bashrc`
2. Verify the sentinel block is present and contains the correct `module load` command.
3. Check if the `.lua` file exists: `ls /export/apps/custom_modules/`
4. Verify the directory is in MODULEPATH: `module use /export/apps/custom_modules && module avail`

### Module not found after loading the metamodule
The metamodule lists a module that doesn't exist in Spack. Run `module avail` on the master node to confirm the exact module name and version, then edit the stack in the UI to correct it.

### Changes not visible on compute nodes
The `/export/apps/custom_modules` directory is served via NFS. If a compute node cannot see the new files, check the NFS mount:
```bash
# On a compute node
df -h | grep export
ls /export/apps/custom_modules/
```
If not mounted, verify the systemd automount unit is active on the compute node.
